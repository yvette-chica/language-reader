A running document summarizing what we've built and the concepts behind it. Good to read before each coding session.

## What We're Building

A Progressive Web App (PWA) for learning languages through audio. Core features:

1. Import audio (URL or file upload) in a target language
2. Auto-generate a timestamped transcript using Whisper
3. Read along while the audio plays (synchronized)
4. Tap words to translate and save them to a vocabulary list

Currently focused on **German** and **Hungarian**. The user's native language (source language) determines what language translations appear in.

## Concepts Covered

### HTTP Requests from the Backend Perspective

You already know how to make HTTP requests from the frontend. On the backend, Express listens for those same requests and decides what code to run based on the URL and method (GET, POST, PATCH, DELETE).

```js
app.get('/api/health', (req, res) => {
  res.json({ status: 'ok' })
})
```

- `req` — the incoming request (contains headers, body, URL params, etc.)
- `res` — the response object — you call methods on it to send something back
- `res.json()` — sends JSON back to the client

### SQLite and Databases

SQLite is a database that lives in a single file on your machine (`language-reader.db`). No separate server to run. When you're ready to host the app, you swap it for PostgreSQL with minimal code changes.

We use a library called `better-sqlite3` to talk to it from Node.

```js
// Query — returns all rows
const courses = db.prepare('SELECT * FROM courses').all()

// Query — returns one row
const user = db.prepare('SELECT * FROM users WHERE email = ?').get(email)

// Insert/update/delete
db.prepare('INSERT INTO users (email) VALUES (?)').run(email)
```

**Important:** Always use `?` placeholders instead of putting variables directly in SQL strings. Putting variables directly in SQL strings is how SQL injection attacks happen — a hacker can craft input that breaks out of your query and runs their own SQL.

**Foreign keys** link rows in different tables (e.g. a course belongs to a user). SQLite has foreign key enforcement *off* by default, so we turn it on explicitly:

```js
db.pragma('foreign_keys = ON');
```

**`ON DELETE CASCADE`** — if you delete a user, all their courses are automatically deleted too. The deletion cascades down the chain.

### REST API Conventions

REST is a convention for how to structure your API URLs and methods:

| Action | Method | URL example |
|---|---|---|
| Get all | GET | `/api/courses` |
| Get one | GET | `/api/courses/1` |
| Create | POST | `/api/courses` |
| Update | PATCH | `/api/courses/1` |
| Delete | DELETE | `/api/courses/1` |

HTTP status codes tell the client what happened:
- `200` — OK
- `201` — Created (used after successful POST)
- `400` — Bad request (missing required fields)
- `401` — Unauthorized (not logged in)
- `403` — Forbidden (logged in but not allowed)
- `404` — Not found
- `409` — Conflict (e.g. email already in use)

### Authentication — Passwords and Hashing

You never store passwords as plain text. If your database is ever compromised, all user passwords would be exposed.

Instead you store a **hash** — a one-way scrambled version of the password using a library called `bcrypt`:

```js
// On register — hash before storing
const password_hash = await bcrypt.hash(password, 12)
// 12 is the "cost factor" — higher = slower to compute = harder to brute force

// On login — compare submitted password to stored hash
const match = await bcrypt.compare(password, user.password_hash)
```

You can never "unhash" a password — you just compare hashes. If they match, the password is correct.

### Authentication — JWT Tokens (Staying Logged In)

HTTP is stateless — the server doesn't remember anything between requests. After login, the frontend needs a way to prove who it is on every subsequent request. We use **JWT (JSON Web Token)** for this.

**The flow:**
1. User logs in with email + password
2. Server checks the password
3. If correct, server generates a signed token containing the user's ID
4. Token is sent back to the frontend
5. Frontend stores the token and sends it with every future request:
   ```
   Authorization: Bearer <token>
   ```
6. Server reads and verifies the token to know who the user is

**Generating a token:**
```js
const token = jwt.sign(
  { id: user.id, email: user.email },  // payload — data encoded in the token
  process.env.JWT_SECRET,               // secret key used to sign it
  { expiresIn: '7d' }                   // token expires after 7 days
)
```

A JWT looks like three base64 strings separated by dots:
```
eyJhbGciOiJIUzI1NiJ9.eyJpZCI6MX0.abc123...
[    header    ].[  payload  ].[signature]
```

The signature is generated using the secret key — so only your server can verify that a token is genuine. If someone tampers with the payload, the signature won't match and the token is rejected.

**The secret key** lives in `.env` and must never be committed to git.

---

### Environment Variables (.env)

Secrets like API keys and JWT secrets must not be in your code (especially not in git). They live in a `.env` file:

```
JWT_SECRET=your-secret-here
DEEPL_API_KEY=your-key-here
```

The `dotenv` library loads these into `process.env` when the server starts:

```js
import 'dotenv/config'  // must be the first import

process.env.JWT_SECRET  // available anywhere after this
```

The `.env` file is listed in `.gitignore` so it's never committed. The `.env.example` file (which is committed) shows what variables are needed without revealing their values.
