## Concepts Covered

### Middleware

Middleware is a function that runs between receiving a request and sending a response. Express runs middleware in order вЂ” each one can modify the request/response or stop the chain early.

```js
// Middleware signature вЂ” always (req, res, next)
function requireAuth(req, res, next) {
  // do something...
  next() // pass the request along to the next handler
}

// Applied to a single route
router.get('/', requireAuth, (req, res) => { ... })
```

You've already used middleware: `express.json()` parses the request body before your route handler sees it. `requireAuth` does the same idea вЂ” it checks the token before your route handler runs.

### bcrypt.compare

On login you can't re-hash the submitted password and compare strings вЂ” bcrypt hashes include a random salt, so the same password produces a different hash each time. Instead:

```js
const valid = await bcrypt.compare(plainTextPassword, storedHash)
// returns true if they match, false if not
```

bcrypt knows how to extract the salt from the stored hash and use it to verify the submitted password.

---

### Why login and register both return the same "Invalid email or password" error

Returning different errors for "email not found" vs "wrong password" would let an attacker figure out which emails have accounts. By returning the same message for both cases, you don't leak that information.

### Nested Routes and URL Parameters

Lessons live inside courses, so their URLs are nested:

```
GET  /api/courses/:courseId/lessons
POST /api/courses/:courseId/lessons
```

`:courseId` is a URL parameter вЂ” Express captures whatever value is in that position and makes it available as `req.params.courseId`.

When using nested routers, you need `mergeParams: true` so the child router can access params from the parent route:

```js
const router = Router({ mergeParams: true })
// req.params.courseId is now available inside this router
```

---

### Ownership checks

When a route acts on a resource (like lessons), you need to verify the parent resource belongs to the logged-in user. A single query can do both the lookup and the check:

```js
db.prepare('SELECT * FROM courses WHERE id = ? AND user_id = ?').get(courseId, userId)
```

If the course doesn't exist *or* belongs to someone else, this returns `undefined`. We return a 404 either way вЂ” we don't want to confirm to a user that a course exists if it's not theirs.
